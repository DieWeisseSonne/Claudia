# Salience-Gated Memory Architecture Spec

> Prior knowledge biases current attention (what enters the workspace) →
> current attention biases future memory (what gets consolidated) →
> future memory biases future attention. The agent develops a *taste*
> for what matters in its deployment context.

This loop is the core of the architecture. Everything else — the tiers,
the PE routing, the heartbeat, the conservative kernel — either feeds
this loop or prevents it from collapsing.

Taste is not a feature you build. It is a *consequence* of a capacity-
limited workspace. The system has to choose what to attend to, and what it
attended to yesterday biases what it attends to today. An agent that
spends its first week on SRE incidents develops richer L3 patterns for
infrastructure failures, which makes infrastructure signals more
recognizable in the workspace, which makes infrastructure knowledge more
likely to be consolidated. It didn't choose to specialize. The loop chose
for it.

The cold start is therefore formative, not just inconvenient. Before the
loop has turned a few times, the agent has no taste — no L3 to bias
attention, so attention is determined by the base model's priors. The
first few sessions seed the loop's initial direction. What the agent
encounters early matters disproportionately.

L4 (the conservative kernel) is the anchor that prevents the loop from
drifting arbitrarily. Without it, taste can be redirected under pressure
— the agent develops obsession rather than expertise. L4 says: regardless
of what the loop discovers, these commitments hold. It is the difference
between an adaptive agent and a runaway one.

## 1. What frontier agents already do

The cognitive architecture is not built from scratch. Frontier agents —
Claude Code, Codex, Letta — have already implemented crude versions of
several core mechanisms. The contribution is not new components but new
*wiring*: connecting compaction, memory, and context management into a
system that produces the feedback loop.

### 1.1 Claude Code's compaction = workspace compression

Claude Code manages context through three-layer compaction:

- **Microcompaction:** offloads bulky tool results to disk, keeps a "hot
  tail" of recent results inline. This is the L0/L1 split — workspace
  vs. session buffer.
- **Auto-compaction:** triggered when free context space drops below
  thresholds. This is the capacity limit forcing integration — GWT
  ignition implemented as context engineering.
- **Manual compaction:** at task boundaries, with focus hints. Compaction
  preserves task lists, restores recent files, injects continuation
  instructions. The compacted state is a model of what the system is
  attending to — the attention schema (Principle 5).

Anthropic's own research confirms "context rot" — accuracy degrades as
context fills. More context is not better. This is the empirical
validation of the capacity-limit principle.

**What Claude Code doesn't do:** the compaction is stateless. It doesn't
learn what to keep. It doesn't route based on surprise. It doesn't feed
back into future compaction decisions. The salience gate adds the loop.

### 1.2 Codex's async memory = post-response consolidation

Codex generates memories asynchronously after threads become idle — not
during active work. Memory updates are skipped when rate-limited. This is
Phase 3 / idle heartbeat consolidation.

Codex also uses prompt caching via exact prefix extension — each prompt
is structured so prior context is reusable. This is an inference
optimization the workspace should adopt.

Codex has demonstrated 25+ hour runs processing 13M tokens. The key is
sustained context management, not single-shot intelligence.

**What Codex doesn't do:** memories are generated by summarization, not
by PE-routed consolidation. There's no distinction between "this extends
what I know" and "this is genuinely new." No strength-based resistance.
No feedback from retrieval utility to future storage decisions.

### 1.3 Letta's tiered storage = memory architecture

Letta (MemGPT) implements agent-managed tiered memory: core memory (in
context, like RAM), recall memory (searchable conversation history), and
archival memory (long-term vector-indexed cold storage). The agent
explicitly decides what to evict, summarize, or archive.

Letta also introduced context repositories — git-backed memory with
versioning and concurrent writes across sub-agents. And "sleep-time
compute" for offline learning during idle periods.

**What Letta doesn't do:** the agent manages memory through explicit
tool calls (push/pull), not as a byproduct of capacity-limited
integration. The salience judgment is decoupled from reasoning. In our
architecture, the salience judgment IS the reasoning — what the narrator
preserves under pressure is what the system considers important.

### 1.4 The production memory taxonomy (2026)

The agent ecosystem has converged on three memory types:

- **Episodic:** timestamped events in vector stores (append-only, inline
  writes). Maps to our L2.
- **Semantic:** deduplicated facts in knowledge graphs (async update).
  Maps to our L3. Zep/Graphiti is the leading implementation — a
  temporal knowledge graph that tracks when facts were learned and
  changed. Already handles temporal reasoning natively.
- **Procedural:** verified tool-call sequences in skill registries
  (write after success). Missing from our current design — see §1.5.

Critical production pattern: async writes. Episodic writes are inline;
semantic and procedural writes are queued to avoid latency. This matches
our Phase 3.

### 1.5 What we're missing: procedural memory

Skill-learning research (SkillCraft, H-EPM, ToolMem) shows that agents
caching verified tool-use sequences get 50–80% token reduction and 40%+
improvement on out-of-distribution tasks.

The architecture needs a procedural dimension — not a separate tier, but
an aspect of L3. Semantic patterns include not just facts ("service X
is unreliable") but action sequences ("when service X is unreliable, the
successful remediation pattern is: check logs → restart pod → verify
health"). These are consolidated from L2 episodes where the same action
sequence succeeded repeatedly.

### 1.6 What's genuinely novel (our contribution)

Frontier agents have solved the component engineering. The five things
none of them do:

1. **The consolidation gate.** Compaction/memory is driven by
   summarization or explicit management. Nobody routes storage decisions
   by prediction error — update existing knowledge vs. create new entry
   based on surprise relative to what's already known.

2. **The feedback loop.** No production agent has a connection between
   what enters the workspace and what gets consolidated that improves
   with experience. Compaction is stateless. Our RL-learned
   consolidation policy is the missing piece.

3. **Temporal binding as agent capability.** Codex and Claude Code have
   timestamps but no temporal reasoning over their own history. Zep's
   temporal graph handles it at the infrastructure level but the agent
   can't introspect on its own temporal experience.

4. **The conservative kernel.** No production agent has write-protected
   identity-core memories with strength-based resistance. System prompts
   are static configuration, not dynamic commitments that earned their
   protection through consolidation strength.

5. **Reconstructive recall.** Everyone pastes retrieved content verbatim.
   Nobody generates from compressed cues, which means nobody gets schema
   distortion (memories shaped by current understanding) — a feature
   that produces context-adaptive recall.

## 2. Design Principles

Seventeen empirical implications from cognitive science, compressed to
their design consequences. These tell us HOW to wire the existing
components to produce the feedback loop. Source experiments are in
`cognitive-design-notes.md`.

### From GWT (Global Workspace Theory)

1. **The workspace gate is binary, not graded.** Under tight capacity
   pressure, the narrator's compression approaches binary behavior.
   The capacity limit IS the ignition mechanism. Claude Code's
   compaction already does this — we make it principled.

2. **One serial integration per cycle.** The workspace processes one
   coherent integration at a time. Buffered inputs wait for the next
   cycle (attentional blink). This is the ReAct loop, constrained.

3. **Integration triggered by novel input, not maintenance.** The
   narrator call fires on new inputs. Between triggers, the workspace
   persists. The heartbeat triggers maintenance, not re-integration.
   Codex's idle memory generation already follows this pattern.

4. **Threshold modulated top-down.** What survives compression depends
   on both input salience AND the current workspace state (goals, prior
   knowledge). Prior knowledge biases what enters. This is precision
   weighting — and it's what creates the feedback loop.

### From AST (Attention Schema Theory)

5. **The workspace state IS the attention schema.** Claude Code's
   compaction already does this — preserving task lists, restoring
   context, injecting "here's where you are." We make it explicit: the
   compressed state is a self-model of attention, readable by
   specialists for coordination.

6. **The schema should emerge, not be hardwired.** Minimal scaffold +
   free-form narrator state. The structure develops through experience.
   No fixed JSON schema for the workspace.

7. **Graceful degradation.** If the workspace is impaired, specialists
   still function. Responses lack integration but the system doesn't
   crash.

### From PP (Predictive Processing)

8. **PE magnitude routes consolidation.** Small PE → UPDATE existing
   memory. Large PE → CREATE new entry. This is the core novel
   mechanism — the consolidation gate.

9. **Two novelty types route to different stores.** Common novelty →
   L3 semantic (including procedural patterns). Distinct novelty →
   L2 episodic.

10. **Strong memories resist modification.** Consolidation strength
    scales with retrieval utility. L4 = maximum strength,
    write-protected. The conservative kernel as a continuum.

11. **Novelty boosts temporally adjacent consolidation.** Surprising
    events boost consolidation of surrounding context.

### From Spens & Burgess (Hippocampo-neocortical RAG)

12. **Store compressed gist, not verbatim.** The workspace output
    under capacity pressure IS the gist. Store it directly as L2.
    This is what Claude Code's compaction produces — we persist it.

13. **Recall is reconstruction, not retrieval.** Generate from
    compressed cues rather than pasting verbatim. Memories adapt to
    current context. This is the one technique no frontier agent uses.

14. **Consolidation = predictive sufficiency.** An L2 episode is
    consolidated to L3 when the system can predict it from L3 alone.
    Letta's sleep-time compute is the execution mechanism.

### From PFC-BG Gating

15. **Consolidation policy is RL-learned.** v1 starts with heuristics
    (recency, retrieval count, user confirmation). RL refines the
    heuristic weights over time. Codex's async memory generation is
    the execution pattern; we add the reward signal.

16. **Capacity scales with expertise.** Effective workspace capacity
    increases as the system learns to compress familiar content.
    Experienced agent in familiar domain fits more into the workspace.

17. **Exploit/update mode switching.** Low accumulated PE = cheap
    consolidation. High accumulated PE = expensive re-evaluation.

## 3. Components

Four components. No Critic, no entity registry, no orchestrator. Built
on top of frontier agent patterns, not from scratch.

### 3.1 Workspace (L0)

Claude Code's compaction system, made principled. A capacity-limited
context region that forces integration under compression pressure.

- **Capacity-limited.** Token budget on the workspace state (tunable,
  default 2–4k tokens). Forces the narrator to choose what survives.
  Claude Code does this implicitly with auto-compaction thresholds;
  we make the limit explicit and structural.
- **Write-gated.** Only the narrator writes. Specialists propose.
- **Broadcast.** All specialists read the workspace state.
- **Temporally indexed.** Heartbeat timestamp on each state.

Format — minimal scaffold + free-form state:

```
workspace = {
  heartbeat: { timestamp, elapsed_since_last, session_id },
  goal: string,
  confidence: float,
  active_items: int,
  state: string    // free-form, capacity-limited
}
```

### 3.2 Narrator

The sole integration point. Functionally equivalent to the main LLM call
in any agent loop (Claude Code's query loop, Codex's agent loop, Letta's
agent step). The difference: it writes to a capacity-limited workspace
rather than an unbounded context window.

1. Receives warm-up context + specialist outputs + input.
2. Compresses under workspace capacity limit → new workspace state.
3. Produces user-facing response.
4. What it preserves under capacity pressure = salience judgment.

### 3.3 Specialist sub-agents

Standard tool calling or sub-agent invocations. Event-triggered,
domain-scoped, ephemeral. They read the workspace broadcast and return
structured results. They do not write to memory or workspace.

No Critic specialist. Metacognition emerges from the narrator
encountering its own prior outputs under capacity pressure (HOT
deflation). The Critic ablation (A3) tests this empirically.

Specialist roster is deployment-specific.

### 3.4 Tiered memory

| Tier | Name | Built on | Content |
|---|---|---|---|
| L0 | Workspace | Claude Code compaction model | Compressed integration state |
| L1 | Session buffer | Codex conversation state / LangGraph checkpoints | Rolling buffer of recent workspace states |
| L2 | Episodic store | Vector store with timestamps | Compressed gist per episode |
| L3 | Semantic store | Temporal knowledge graph (Zep/Graphiti pattern) + procedural patterns | Extracted facts, relationships, and action sequences |
| L4 | Conservative kernel | System prompt + persistent KV store | Identity-core commitments, write-protected |

**L1** is a buffer, not a gated tier. Everything the narrator produces
goes to L1 automatically. The gate is L1 → L2.

**L2 entry format:**

```
{
  id: uuid,
  timestamp: heartbeat_timestamp,
  session_id: int,
  gist: string,
  goal_at_time: string,
  embedding: vector,
  source: enum(user, tool, inference, reconstruction),
  strength: float,
  retrieval_count: int,
  last_retrieved: timestamp
}
```

**L3** includes both semantic (facts, entity states, relationships) and
procedural (verified action sequences that succeeded). Procedural entries
are consolidated from L2 episodes where the same pattern succeeded
repeatedly — the skill-learning pathway.

**L4 as living influence.** L4 entries are associated with bias agents
that activate during warm-up. When input arrives, L4 bias agents check
whether the input touches kernel-level concerns and inject flags into
the warm-up context. Deep, constant, low-bandwidth influence.

## 4. The Salience Gate

The single mechanism from which four capacities emerge: what to attend
to, what to learn, what to become, what to trust. One bottleneck, four
capacities. This is the core novel contribution — the wiring that no
frontier agent has.

### 4.1 Two-stage gate

**Stage 1 — Attention gate (what enters the workspace).**
The narrator compresses under capacity pressure. What survives
compression has been judged worth attending to. The capacity limit IS
the gate. This is Claude Code's compaction made into a salience
mechanism.

**Stage 2 — Consolidation gate (what leaves into memory).**
Not everything attended to is worth remembering. The consolidation gate
(Phase 3) judges cross-episode utility. This is Codex's async memory
generation made into a PE-routed decision.

### 4.2 PE computation

Piggybacks on retrieval. No extra LLM call.

1. Embed the workspace output.
2. Search L2/L3 for nearest neighbors (already needed for retrieval).
3. PE = 1 − max_cosine_similarity.
4. PE > threshold_high → new L2 entry (accommodation).
5. PE < threshold_low → update nearest L3 entry (assimilation).
6. Between thresholds → new L2 entry (distinct but relatable).

### 4.3 Consolidation routing

- **Small PE, extends existing pattern:** update L3 (semantic or
  procedural). Common novelty refines existing expertise.
- **Large PE, no existing pattern:** encode into L2. Distinct novelty.
- **L2 → L3 promotion (idle):** when L2 accumulates related episodes,
  extract the shared pattern into L3. This is Letta's sleep-time
  compute with PE-informed clustering.

### 4.4 Consolidation strength

```
strength = (retrieval_count * retrieval_utility_avg)
         + age_bonus(days_since_creation)
         - disuse_penalty(days_since_last_retrieval)
         + user_confirmation_bonus
```

Update threshold scales with strength:
`PE_required_to_modify = base_PE * (1 + strength)`.

L4 promotion: strength exceeds hard threshold → write-protected,
always-loaded.

### 4.5 Temporal boost (STC mechanism)

Rolling buffer of last N workspace outputs (from L1) with heartbeat
timestamps. When PE > threshold_high: retroactively flag last K buffer
entries for L2 promotion. Boost decays with temporal distance.

### 4.6 The feedback loop and its risks

The loop: prior knowledge → current attention → future memory → future
attention. The agent develops taste.

**Salience collapse** — the loop closes too tight:
- Novelty bonus in consolidation (high PE = high learning value).
- Exploration budget in the attention gate.
- Periodic memory audit on heartbeat.
- Human-in-the-loop override.

## 5. Reactive-with-Warmup Execution Model

The production execution model. Matches the patterns frontier agents
already use — Claude Code's input governance, Codex's async memory,
Letta's idle consolidation — wired through the salience gate.

### Phase 1 — Warm-up (cheap, parallel, no narrator call)

1. Embed input, retrieve from L2/L3.
2. Load relevant L4 kernel entries.
3. Activate L4 bias agents — flag kernel-level concerns.
4. Pull recent workspace states from L1 rolling buffer.
5. Trigger relevant specialists (parallel execution).

This is Claude Code's "input governance" sequence: memory prefetch,
skill discovery, context assembly — before the model ever sees the input.

### Phase 2 — Integration (one LLM call)

6. Narrator receives: warm-up context + specialist outputs + input.
7. Compress under workspace capacity limit → new workspace state.
8. Produce user-facing response.
9. Broadcast workspace state.

This is the main agent loop call — same as Claude Code's query loop,
Codex's model invocation, Letta's agent step. The difference: the output
is a capacity-limited workspace state, not an unbounded context append.

### Phase 3 — Post-response (async)

10. Embed new workspace.state.
11. PE computation (cosine distance to nearest L2/L3 neighbors).
12. Consolidation routing (L1 → L2 decision).
13. Temporal boost check.
14. Strength updates for entries used in step 1.

This is Codex's async memory generation, made into a PE-routed
consolidation decision rather than a summarization.

### Idle heartbeat

15. L2 → L3 consolidation (cluster episodes, extract patterns — both
    semantic and procedural).
16. Memory decay (reduce strength of unretrieved entries).
17. Memory audit (distribution balance).
18. Novelty countermeasures (salience collapse detection).

This is Letta's sleep-time compute, directed by PE-informed clustering.

### Latency budget

User-facing latency = Phase 1 + Phase 2. Phase 3 uses the LIVE workspace
state — the salience signal is still fresh. Idle heartbeat operations
run deferred.
